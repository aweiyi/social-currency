# FavourNet v2 — Updated Architecture

Social Token Simulation — NS Hackathon Project Plan
Project Name: FavourNet — Social Currency Simulation for Network School
One-liner
A browser-based simulation showing how social tokens (favour-based currency) flow through a Network School community, what behaviours they incentivise, and how this functions alongside fiat currency economy in Network School. 

CONCEPT SUMMARY
FavourNet simulates a micro-economy inside Network School where members issue personal social tokens instead of using money. Each token represents a promise of future value (a favour).

Three core mechanisms:

Proof of Contribution (PoC) — attestation system that builds reputation
Social Token Exchange — personal tokens issued, traded, redeemed at market prices
Community Insurance Pool — mutual fund that absorbs defaults

## 1. GENERATIVE AGENTS — HYBRID APPROACH

### Why not full Stanford-style generative agents?

The Stanford generative agents paper runs LLM calls per agent per tick
for memory retrieval, reflection, and planning. With 180 agents, that's
180+ API calls per simulation tick. Even with Groq's free tier
(~30 requests/minute on free plan), you'd hit rate limits within
seconds and each tick would take minutes to process.

### The hybrid approach (feasible for hackathon):


### Agent generation script (run before hackathon):

```javascript
// scripts/generateAgents.js
// Run: node scripts/generateAgents.js

const GROQ_API_KEY = "your-free-groq-key"; // get from console.groq.com
const BATCH_SIZE = 20;
const TOTAL_AGENTS = 180;
const BATCHES = Math.ceil(TOTAL_AGENTS / BATCH_SIZE); // 9 batches

async function generateBatch(batchNum) {
  const response = await fetch("https://api.groq.com/openai/v1/chat/completions", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "Authorization": `Bearer ${GROQ_API_KEY}`
    },
    body: JSON.stringify({
      model: "llama-3.3-70b-versatile",
      messages: [{
        role: "user",
        content: GENERATION_PROMPT.replace("{BATCH_SIZE}", BATCH_SIZE)
          + `\n\nThis is batch ${batchNum + 1} of ${BATCHES}. Ensure names don't repeat.`
      }],
      temperature: 0.9,  // high creativity for diversity
      max_tokens: 4000
    })
  });

  const data = await response.json();
  const text = data.choices[0].message.content;
  // Strip markdown fences if present
  const clean = text.replace(/```json|```/g, "").trim();
  return JSON.parse(clean);
}

async function generateAll() {
  const allAgents = [];
  for (let i = 0; i < BATCHES; i++) {
    console.log(`Generating batch ${i + 1}/${BATCHES}...`);
    const batch = await generateBatch(i);
    allAgents.push(...batch);
    // Respect rate limits — wait 3 seconds between batches
    await new Promise(r => setTimeout(r, 3000));
  }

  // Add IDs and initialise token state
  const agents = allAgents.slice(0, TOTAL_AGENTS).map((a, i) => ({
    id: `agent_${String(i + 1).padStart(3, "0")}`,
    ...a,
    reputation: {
      overall: a.personality.reliability * 0.5 + 0.3, // initial estimate
      byCategory: {}
    },
    tokens: {
      issued: 0,
      redeemed: 0,
      held: [],
      weeklyCapacity: 10 + Math.floor(a.personality.social_drive * 5),
      remainingCapacity: 10 + Math.floor(a.personality.social_drive * 5)
    },
    fiatBalance: 1000, // everyone starts with 1000 fiat units
    marketPrice: a.personality.reliability * 0.5 + 0.3,
    isActive: true,
    redemptionSpeed: a.personality.reliability * 0.7 + 0.3
  }));

  // Write to file
  const fs = require("fs");
  fs.writeFileSync(
    "src/data/seedAgents.json",
    JSON.stringify(agents, null, 2)
  );
  console.log(`Generated ${agents.length} agents → src/data/seedAgents.json`);
}

generateAll().catch(console.error);
```

## 3. DUAL CURRENCY MODEL — TOKENS ALONGSIDE FIAT

### Core principle: "When presented with a choice of favours vs money, choose favours"

Social tokens don't replace fiat. They create a **parallel economy**
where certain exchanges are better served by social currency. The
simulation models BOTH and shows when/why agents choose one over
the other.

### Decision logic for each exchange:

```javascript
function decidePaymentMethod(requester, provider, category, hours) {
  const fiatCost = hours * SKILL_CATEGORIES[category].fiatEquivalent;
  const tokenCost = hours; // in contribution-hour tokens
  const tokenValue = tokenCost * requester.marketPrice; // market value

  // Factors favouring social tokens:
  const socialScore =
    (1 - requester.fiatPreference) * 0.3 +         // personal preference
    (1 - provider.fiatPreference) * 0.2 +           // provider's preference
    requester.personality.social_drive * 0.2 +       // social orientation
    (requester.reputation.overall > 0.7 ? 0.2 : 0) + // high rep = tokens accepted
    (fiatCost > requester.fiatBalance * 0.1 ? 0.1 : 0); // low on fiat

  // Factors favouring fiat:
  const fiatScore =
    requester.fiatPreference * 0.3 +
    provider.fiatPreference * 0.2 +
    (requester.marketPrice < 0.5 ? 0.3 : 0) +       // low market price = tokens less accepted
    (category === "transport_errands" ? 0.1 : 0) +   // quick transactions favour fiat
    (requester.tokens.remainingCapacity < 2 ? 0.1 : 0); // near capacity limit

  if (socialScore > fiatScore) {
    return {
      method: "social_token",
      amount: tokenCost,
      fiatEquivalent: fiatCost,
      reasoning: "favour-based exchange"
    };
  } else {
    return {
      method: "fiat",
      amount: fiatCost,
      reasoning: "monetary exchange"
    };
  }
}
```

### What this reveals in simulation:

1. **Token/fiat ratio over time** — Does the community naturally shift
   toward more social token usage as trust builds? Or does fiat
   dominate when reputation is uncertain?

2. **Category preferences** — Do certain types of favours naturally
   gravitate toward tokens (mentorship, pet care, cooking) while
   others stay fiat (transport, quick tasks)?

3. **Social stratification** — Do high-reputation people operate
   primarily in the token economy while low-reputation people are
   "stuck" in fiat? Is that a feature or a problem?

4. **Fiat drainage** — In a closed community, where does fiat go?
   Does it accumulate with certain people? Do social tokens
   redistribute more evenly?

5. **The "choose favours" thesis** — Under what conditions do agents
   actually prefer favours over money? Your hypothesis is that this
   is the rational choice in a trust-rich community. The simulation
   can test that.

### The "Social Premium" metric:

This is your killer insight for the demo. Define:

```
Social Premium = (total value exchanged via tokens) / (fiat equivalent of same exchanges)
```

If Social Premium > 1.0, it means social tokens are generating MORE
value than fiat would have — because they unlock exchanges that
wouldn't have happened with money alone (you wouldn't pay someone
$60/hr for advice, but you'd happily exchange a favour).

If Social Premium < 1.0, it means the overhead of reputation tracking
and trust maintenance is destroying value.

Your thesis predicts Social Premium > 1.0 in a tight community like NS.
The simulation can show under what conditions this holds.

### Pricing Formula (v2)

function calculateMarketPrice(agent, allAgents) {
  const reputation = agent.reputation.overall;                    // 0-1
  const obligationRatio = Math.min(agent.tokens.issued /
    Math.max(agent.tokens.redeemed, 1), 5) / 5;                  // 0-1 (capped)
  const velocity = agent.redemptionSpeed;                         // 0-1
  const scarcity = calculateCategoryScarcity(agent, allAgents);   // 0-1

  return (
    reputation * 0.40 +
    (1 - obligationRatio) * 0.25 +  // lower ratio = higher price
    velocity * 0.20 +
    scarcity * 0.15
  );
}

function calculateCategoryScarcity(agent, allAgents) {
  // For agent's top skill, how rare is it in the community?
  const topSkill = getTopSkill(agent);
  const providersCount = allAgents.filter(a =>
    a.skills[topSkill] > 0.5 && a.isActive
  ).length;
  return 1 - (providersCount / allAgents.length);
}

COMMUNITY POOL MECHANISM
Funding

Every token issuance: 10% levy goes to community pool
Levy is allocated to the token's skill category in the pool
Example: Agent issues 2.0 software tokens → 0.2 hours added to pool's software capacity

Default Handling
When an agent defaults (leaves or can't deliver):

All their outstanding tokens are marked "defaulted"
Defaulted tokens enter a 7-tick absorption window
Any active agent in the same skill category can volunteer to absorb
Volunteer gets: the original obligation + 1.5x reputation bonus
If no volunteer after 7 ticks: pool covers what it can (up to pool's category balance)
Any remaining uncovered amount is written off (token holders lose value)
Defaulter's remaining tokens across all holders are repriced to 0

### Pool Health
poolHealthScore = poolCapacity / totalOutstandingObligations
// > 0.8 = healthy (green)
// 0.5-0.8 = caution (yellow)
// < 0.5 = stressed (red) — system is under-collateralised

SIMULATION LOOP (each tick = ~1 simulated day)
1. GENERATE NEEDS
   - Each agent has a random chance of needing help in a category
   - Need probability weighted by inverse of their own skill
     (you need help with things you're bad at)

2. MATCH COUNTERPARTIES
   - Agent with need searches for provider: highest reputation
     in needed category, who has remaining issuance capacity
   - If match found → proceed to token exchange

3. ISSUE TOKENS
   - Requester issues tokens to provider
   - Amount = estimated hours (randomised 0.5-3.0)
   - 10% levy deducted to community pool
   - Provider's held tokens updated, requester's issued count updated

4. PROCESS REDEMPTIONS
   - Each tick, some held tokens are randomly redeemed
   - Redemption speed influenced by issuer's redemptionSpeed stat
   - On redemption: both parties get PoC attestation boost

5. HANDLE DEFAULTS
   - Small random chance (2%) an agent becomes inactive each tick
   - Triggers default cascade for all their outstanding tokens
   - Pool absorption mechanism activates

6. UPDATE PRICES
   - Recalculate marketPrice for all active agents
   - Based on updated reputation, obligation ratios, velocity, scarcity

7. UPDATE METRICS
   - Gini coefficient of token holdings
   - Pool health score
   - Total circulation volume
   - Category-level supply/demand balance

8. LOG EVENTS
   - Push notable events to event feed for the dashboard

## 8. PERFORMANCE NOTES FOR 180 AGENTS

With 180 agents vs 25, we need to watch performance:

- **Simulation tick**: O(n²) matching is fine for 180 (32,400 comparisons).
  JavaScript handles this in <1ms.
- **React rendering**: Don't render all 180 agents simultaneously.
  Use virtualised list (show 20, scroll for more) or pagination.
- **Charts**: Recharts with 180 lines would be unreadable.
  Price chart: show top 10 agents only (highest volume).
  Other charts aggregate, so 180 is fine.
- **History arrays**: Cap at last 200 ticks to prevent memory bloat.
  At 200 ticks × ~30 metrics, that's ~6000 data points — no problem.
- **Event feed**: With ~30 exchanges/tick, only show "notable" events.
  Filter: defaults, pool events, price swings >10%, large trades.
  This keeps the feed useful rather than a blur.
